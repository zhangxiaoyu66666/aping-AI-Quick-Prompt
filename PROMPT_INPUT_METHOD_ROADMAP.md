# 啊拼 / AI Quick Prompt 产品路线图

更新时间：2026-06-02

这份文档记录啊拼 1.0 公开发布之后的产品与技术路线。它不是 1.0 发布验收清单；已完成的发布审查、OCR 许可证审查、模板归属和打包检查请看 [docs/release-1.0-readiness.md](docs/release-1.0-readiness.md)。

## 路线原则

- **Windows 体验先稳定**：啊拼 1.0 是 WinUI 3 + .NET 8 Windows 桌面应用，后续仍优先修稳定性、性能、可访问性和本地化。
- **跨平台先拆核心，再做外壳**：macOS 不能直接复用 WinUI 3。要先把提示词路由、模板、Skill、Provider、OCR 调度等核心逻辑从 WinUI UI 层拆得更清楚，再决定 macOS 使用 Avalonia、.NET MAUI、SwiftUI 或其他方案。
- **本地优先，云服务分支隔离**：GitHub GPL 分支保持本地优先。微软商店分支可以保留独占云服务能力，但不混入公开路线图作为默认能力。
- **不把上游项目源码树塞进 Git**：继续保留生成后的模板数据、来源标签、许可证和归属信息；复制的维护参考树留在本地。
- **模型资产先审查后发布**：OCR 模型、原生依赖和转换产物必须有来源、许可证、哈希和声明记录后才能进入公开发布包。

## 1.0.x：Windows 稳定维护

目标：让 1.0 成为可靠的日常工具，而不是只会启动的演示版。

- 修复托盘退出、窗口关闭、快捷键恢复、置顶小窗和多进程残留问题。
- 继续压缩模板库加载卡顿，确保模板切换、Skill 切换和常用提示词菜单不会阻塞 UI 线程。
- 完善 Fire Eye OCR 默认路径，Windows Media OCR 只作为 worker 不可用时的兜底方案。
- 调整截图 OCR 等待时间，避免动画残影、半透明窗口残影进入截图。
- 优化深度思考流式输出，确保思考过程气泡、打字机效果和最终提示词输出互不阻塞。
- 统一中文界面文案，消除中英文混杂；英文界面由官方 `.resw` 本地化资源提供。
- 增强小窗口模式，让输入需求、Skill、优化目标和聊天式追问与大窗口行为一致。
- 保持 MSIX 侧载包、公开 demo zip 和 GitHub Release 的发布流程可重复。

## 1.1：Windows 体验增强

目标：把“能用”打磨成“顺手”。

- 将快速提示词模板从磁贴列表进一步改为高效下拉菜单，模板与 Skill 不混在同一个入口里。
- 增强常用提示词收藏，支持从中文提示词框和英文提示词框分别收藏到常用提示词。
- 补齐键盘导航、屏幕阅读器名称、焦点顺序和高 DPI 布局测试。
- 继续优化 Acrylic 托盘菜单、小窗动画、导航切换动效和 Fluent 视觉一致性。
- 增加更多截图 UI 回归检查，覆盖展开、紧凑、窄版、高 DPI 和托盘菜单。
- 改进 Provider 能力识别，用内置提供商标签判断深度思考、Markdown、多模态和工具调用能力。
- 增强 Markdown 输出控制：支持 Markdown 渲染、去 Markdown、禁止模型输出 Markdown 三种策略。

## 1.2：macOS 技术预研

目标：明确 macOS 适配方案，避免盲目移植。

### 需要拆出的共享核心

- 提示词优化目标、路由和模板选择逻辑。
- Skill 挂载、触发词提取和上下文注入逻辑。
- OpenAI-compatible Provider 配置、验证和模型能力标签。
- 模板导入导出、常用提示词和本地收藏格式。
- 本地隐私策略、脱敏、外发确认和日志边界。
- OCR 结果清洗、上下文整理和 fallback 策略。

### macOS 外壳候选

- **Avalonia**：更接近跨平台桌面统一 UI，适合保留 .NET 生态，但需要重新实现 Fluent/WinUI 视觉细节。
- **.NET MAUI**：适合 .NET 跨平台，但桌面体验和复杂窗口控制需要验证。
- **SwiftUI 原生外壳**：macOS 体验最好，但需要把核心逻辑通过库、服务或进程桥接出来。

### macOS 系统能力调研

- 全局快捷键和菜单栏常驻。
- 剪贴板读取、写回和隐私授权。
- 前台窗口信息、窗口标题和应用 bundle 识别。
- 截图权限、区域截图和 ScreenCaptureKit。
- OCR 路径：Apple Vision OCR、CoreML、Fire Eye OCR / MNN Metal 后端可行性。
- API Key 存储：macOS Keychain。
- 安装与分发：`.dmg`、`.pkg`、notarization、公证和 Gatekeeper。

验收标准：

- 输出一份 macOS 技术选型说明。
- 完成一个最小 macOS 原型：全局快捷键、输入框、模型调用、复制输出。
- 明确哪些能力可以共享，哪些能力必须按平台重写。

## 1.3：macOS Preview

目标：做出可公开试用的 macOS 预览版。

- 支持菜单栏常驻和全局快捷键呼出。
- 支持输入需求、优化目标、模型调用、中文/英文输出。
- 支持常用提示词、模板下拉菜单和基础 Skill 挂载。
- 使用 Apple Vision 或 Fire Eye OCR macOS 后端提供本地 OCR。
- 使用 Keychain 保存 Provider API Key。
- 提供 `.dmg` 或 `.pkg` 预览包。
- 在 README 中明确 macOS Preview 的功能差异和已知限制。

不在 Preview 阶段承诺：

- 完整 WinUI 视觉复刻。
- 与 Windows 版完全一致的托盘菜单、小窗动画和窗口行为。
- 云服务能力。
- 未审查模型资产的再分发。

## 1.4：跨平台模板与 Skill 生态

目标：让模板、Skill 和语言包成为真正可迁移的数据，而不是绑定某个 UI。

- 固化模板、常用提示词、Skill、优化目标和 Provider 标签的数据格式。
- 为模板导入导出增加版本号、来源、许可证和冲突处理。
- 支持语言包挂载和导出，继续使用 WinUI 官方 `.resw` 方案作为 Windows 主路径。
- 增加模板库维护脚本，避免直接提交上游源码树。
- 增加更多自动化检查：提示路由、模板导入导出、Skill 匹配、Provider 能力验证、Markdown 策略。

## 2.0：跨平台正式版

目标：Windows 和 macOS 都具备稳定日常使用能力。

- Windows 版继续保持 WinUI 3 原生体验。
- macOS 版提供独立原生或跨平台外壳。
- 共享核心逻辑保持统一测试覆盖。
- 本地数据格式跨平台兼容。
- Provider、模板、Skill 和语言包可以在不同平台间导入导出。
- OCR 后端按平台选择：Windows 优先 Fire Eye OCR，macOS 优先 Apple Vision / Fire Eye OCR Metal 路径。
- 发布包包含完整许可证、第三方声明和模型资产审查记录。

## 长期方向

- Linux 可行性调研：先验证剪贴板、全局快捷键、截图权限和 OCR 路径，再决定是否进入正式路线。
- 插件化 Provider 能力标签：按模型真实能力选择深度思考、Markdown、多模态、图片生成、视频提示词和代码智能体策略。
- 更强的本地隐私控制：更细粒度外发确认、本地日志审计和敏感内容提示。
- 模板与 Skill 社区共享：只共享文本工作流和归属信息，不执行外部脚本，不默认包含破坏性动作。
- 商店版云服务：保留在微软商店分支或商业分支中，公开 GPL 分支默认不启用。

## 非目标

- 不把啊拼变成夏语桌宠前端移植版。
- 不复制天雷爆炸 / 大爆炸的完整交互。
- 不把未经审查的模型资产、上游源码树或本地用户数据提交到公开仓库。
- 不默认上传 OCR 文本、剪贴板内容、用户提示词或本地收藏。
- 不为了跨平台牺牲 Windows 版当前的原生体验。

## 当前发布基线

1.0 已经完成公开发布需要的基础工作：WinUI 3 Windows 应用、Fire Eye OCR 发布审查、模板归属、轻量测试、公开 demo 包和 MSIX 侧载包。后续路线从这里继续向稳定性、macOS 适配和跨平台数据格式推进。
